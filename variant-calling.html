變異呼叫 (SNV / Indel)

從 BAM 到 VCF——GATK HaplotypeCaller、DeepVariant 與 joint genotyping 的核心邏輯。

From BAM to VCF — the core logic of GATK HaplotypeCaller, DeepVariant, and joint genotyping.

概覽

變異呼叫到底在做什麼？

從 BAM 出發，variant caller 對每個基因組位置回答兩個問題：(1) 這個位置與參考序列不同嗎？(2) 樣本的基因型是 0/0、0/1 還是 1/1？ 答案以 VCF 格式輸出，每行一個位點（site），含位置、ref、alt、品質、過濾、樣本基因型。

背後並非簡單比對：caller 必須區分「真實變異」與「測序錯誤、比對假象、PCR error」。現代主流方法分兩大流派——基於 HMM/概率模型（GATK HaplotypeCaller、bcftools）與基於深度學習（DeepVariant、Clair3）。

Starting from BAM, a variant caller answers two questions per genome position: (1) Does this site differ from the reference? (2) Is the sample 0/0, 0/1, or 1/1? Output is VCF — one site per line, with position, ref, alt, quality, filter, and per-sample genotypes.

It's not simple alignment counting: callers must separate "real variants" from "sequencing errors, mapping artifacts, PCR errors". Two modern schools — HMM/probabilistic (GATK HaplotypeCaller, bcftools) and deep learning (DeepVariant, Clair3).

💡
核心原則：SNV 與 short Indel 一起呼叫；structural variants & CNV 用不同工具（見 Step 7）。Joint genotyping（多樣本一起呼叫）比單樣本 calling 在 rare variant 與 sensitivity 上明顯更好。 Core principle: SNVs and short indels are called together; SVs and CNVs require different tools (see Step 7). Joint genotyping (calling all samples together) significantly outperforms single-sample calling for rare variants and sensitivity.
主流方法

一、主流 SNV/Indel Callers

🧠

GATK HaplotypeCaller

Broad Institute 的 best practice 標準（v4.6.0.0）。邏輯：在 "ActiveRegion" 內局部重新組裝 reads，產生候選 haplotypes，再用 PairHMM 計算每條 haplotype 的可能性，最終 genotype。強項：Indel 處理穩健、GVCF + joint genotyping 適合大型隊列。新發展：Broad x Illumina 已將 DRAGEN 演算法整合進 GATK 4。

Broad Institute's best-practice standard (v4.6.0.0). Logic: within "ActiveRegions", locally reassemble reads into candidate haplotypes, compute haplotype likelihoods via PairHMM, then genotype. Strengths: robust indel handling; GVCF + joint genotyping scales to large cohorts. 2025: Broad×Illumina integrated DRAGEN algorithms into GATK 4.

🤖

DeepVariant

Google 開發的 CNN-based caller（v1.8.0, 2025）。邏輯：把 BAM pileup 轉成圖片，用影像分類網路判斷變異。強項：Indel 與低品質區域準確度領先 GATK；trio Mendelian error rate ~3% vs GATK ~5%；無需 BQSR。弱項：joint genotyping 工具鏈仍以 GLnexus 為主，比 GATK 整合性低。

Google's CNN-based caller (v1.8.0, 2025). Logic: convert BAM pileup to images, classify with image-CNN. Strengths: leads GATK on indels and difficult regions; trio Mendelian error ~3% vs GATK ~5%; no BQSR needed. Weakness: joint-genotyping tooling (GLnexus) is less integrated than GATK.

⚡

bcftools mpileup + call

輕量、快速的傳統選擇。邏輯：逐位置 pileup 計算 allele likelihood + Bayesian genotype。強項：速度、低資源、可 stream；非常適合教學與快速 prototype。弱項：Indel 與複雜區域準確度不如 GATK/DeepVariant；對 cohort 規模不擴展。

Lightweight, fast traditional choice. Logic: per-position pileup with allele likelihood + Bayesian genotyping. Strengths: speed, low resource, streamable; great for teaching and quick prototyping. Weakness: less accurate on indels and complex regions; doesn't scale to cohorts.

🌊

Clair3 / Clair3-RNA (long reads)

ONT/PacBio 長讀長首選（2025 最新 v1.0.11）。結合 pileup 與 full-alignment 兩個 NN 模型；2025-03 加入 sequence head/tail calling 與 haplotype-aware workflow。Clair3-RNA 處理長讀長 RNA-seq。對 ONT 而言，Clair3 + DeepVariant 是 PrecisionFDA Truth Challenge 主流選擇。

Top choice for ONT/PacBio long reads (v1.0.11, 2025). Combines pileup and full-alignment neural networks; 2025-03 added sequence head/tail calling and haplotype-aware workflow. Clair3-RNA handles long-read RNA-seq. For ONT, Clair3 + DeepVariant dominate PrecisionFDA Truth Challenge.

互動模擬

二、Allele Frequency vs Genotype Likelihood

給定該位置 n 條 reads，其中 k 條支持 alt allele。在 binomial model 下，三種基因型 (0/0、0/1、1/1) 的 likelihood 為：
P(data | 0/0) ∝ (1-ε)^(n-k) · ε^k  |  P(data | 0/1) ∝ 0.5^n  |  P(data | 1/1) ∝ (1-ε)^k · ε^(n-k)

其中 ε 為定序錯誤率。拖動下方 alt 比例觀察 caller 如何決定。

Given n reads at the site, k of which support the alt allele, the binomial-model likelihoods for the three genotypes are:
P(data | 0/0) ∝ (1-ε)^(n-k) · ε^k  |  P(data | 0/1) ∝ 0.5^n  |  P(data | 1/1) ∝ (1-ε)^k · ε^(n-k)

where ε is sequencing error rate. Drag the sliders to see how a caller decides.

AF: alt allele frequency

QD: QUAL / DP — < 2 過濾
FS: Strand bias Fisher score — > 60 (SNV) / > 200 (Indel) 過濾

四、Caller 選擇

🌳 變異呼叫器決策樹

Q1:
大型 cohort (n > 100)、joint calling？→ 是 → GATK HaplotypeCaller GVCF + GenomicsDBImport + GenotypeGVCFs
Q2:
單樣本 / 小批次 Illumina germline？→ 是 → DeepVariant（最高 indel 精確度，無需 BQSR）
Q3:
長讀長 (ONT / PacBio HiFi)？→ 是 → Clair3（或 PEPPER-Margin-DeepVariant for ONT phased）
Q4:
somatic（腫瘤）變異？→ 是 → GATK Mutect2（best practice）或 Strelka2、DeepSomatic
Q5:
快速 prototype / 教學？→ 是 → bcftools mpileup + call（最簡單）
Q1:
Large cohort (n > 100), joint calling? → Yes → GATK HaplotypeCaller GVCF + GenomicsDBImport + GenotypeGVCFs
Q2:
Single sample / small batch Illumina germline? → Yes → DeepVariant (best indel precision, no BQSR needed)
Q3:
Long reads (ONT / PacBio HiFi)? → Yes → Clair3 (or PEPPER-Margin-DeepVariant for ONT phased)
Q4:
Somatic (tumor) variants? → Yes → GATK Mutect2 (best practice), Strelka2, or DeepSomatic
Q5:
Fast prototyping / teaching? → Yes → bcftools mpileup + call (simplest)
⚠️
常見錯誤：① 多樣本各自呼叫後 merge VCF（會遺失 rare variant 證據）→ 改用 GVCF + joint genotyping；② RNA-seq 直接用 HaplotypeCaller（必須先 SplitNCigarReads）；③ 沒做 VQSR / hard filtering → 大量 false positives 流入下游。 Common mistakes: ① Calling each sample separately then merging VCFs (loses rare variant evidence) → use GVCF + joint genotyping; ② Feeding RNA-seq directly to HaplotypeCaller (must SplitNCigarReads first); ③ Skipping VQSR / hard filtering → flood of false positives.
程式碼

變異呼叫實作

# --- GATK HaplotypeCaller + GVCF + Joint Genotyping (per-cohort) ---
# 1. 每樣本產 GVCF / Per-sample GVCF
for s in S1 S2 S3; do
  gatk HaplotypeCaller \
       -R ref/hg38.fa \
       -I bam/${s}.recal.bam \
       -O gvcf/${s}.g.vcf.gz \
       -ERC GVCF
done

# 2. 合併到 GenomicsDB / Build GenomicsDB
gatk GenomicsDBImport \
     -V gvcf/S1.g.vcf.gz -V gvcf/S2.g.vcf.gz -V gvcf/S3.g.vcf.gz \
     --genomicsdb-workspace-path my_db \
     --intervals chr22

# 3. Joint genotyping / Joint genotyping
gatk GenotypeGVCFs \
     -R ref/hg38.fa \
     -V gendb://my_db \
     -O cohort.vcf.gz

# 4. VQSR or hard-filter
gatk VariantFiltration \
     -R ref/hg38.fa -V cohort.vcf.gz \
     --filter-expression "QD < 2.0"  --filter-name QD2 \
     --filter-expression "FS > 60.0" --filter-name FS60 \
     --filter-expression "MQ < 40.0" --filter-name MQ40 \
     -O cohort.filtered.vcf.gz
# --- DeepVariant (Docker / Singularity) ---
docker run -v $(pwd):/data google/deepvariant:1.8.0 \
  /opt/deepvariant/bin/run_deepvariant \
    --model_type=WGS \
    --ref=/data/ref/hg38.fa \
    --reads=/data/bam/S1.dedup.bam \
    --output_vcf=/data/vcf/S1.deepvariant.vcf.gz \
    --output_gvcf=/data/vcf/S1.deepvariant.g.vcf.gz \
    --num_shards=16

# Joint genotyping with GLnexus
glnexus_cli --config DeepVariantWGS \
            S1.deepvariant.g.vcf.gz S2.deepvariant.g.vcf.gz \
  | bcftools view -Oz -o cohort.dv.vcf.gz
# --- bcftools (輕量、適合教學) ---
bcftools mpileup -f ref/hg38.fa -Ou bam/S1.recal.bam | \
  bcftools call -mv -Oz -o S1.bcf.vcf.gz
bcftools index -t S1.bcf.vcf.gz

# 基本品質過濾 / Basic QC filter
bcftools filter -e 'QUAL<20 || DP<10' -s LowQual -Oz \
  -o S1.bcf.filtered.vcf.gz S1.bcf.vcf.gz

# 看統計 / Stats
bcftools stats S1.bcf.filtered.vcf.gz | grep ^SN
# cyvcf2 — 高效 VCF 走訪 / Fast VCF iteration
from cyvcf2 import VCF

for v in VCF("cohort.vcf.gz"):
    if v.FILTER is not None: continue      # skip filtered
    if v.is_snp and v.QUAL > 30 and v.INFO.get("DP") > 10:
        ad  = v.format("AD")             # shape: (n_samples, 2)
        gq  = v.format("GQ")
        vaf = ad[:, 1] / ad.sum(axis=1)
        print(v.CHROM, v.POS, v.REF, v.ALT[0], v.QUAL, vaf)

# 取得樣本基因型 / Sample genotypes
v = next(VCF("cohort.vcf.gz"))
print(v.gt_types)   # 0=hom-ref, 1=het, 2=unknown, 3=hom-alt
print(v.gt_phases)  # True if phased (|)
